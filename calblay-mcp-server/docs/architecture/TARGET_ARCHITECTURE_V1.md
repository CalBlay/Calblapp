# Target Architecture V1 (Query Platform)

## Objective

Move from "LLM + tools" to a governed query platform where:

1. intent and slots are interpreted consistently,
2. source-of-truth is resolved deterministically,
3. calculations are deterministic and auditable,
4. answers always include source + confidence + trace,
5. learning loop improves routing quality without breaking correctness.

---

## Core Principles

- Deterministic math, probabilistic interpretation.
- Single source of truth per metric.
- No "final value" generated by the model.
- Every answer must be replayable from trace.
- Release is blocked on business golden tests.

---

## Layered Architecture

## 1) Semantic Layer (Business Metric Catalog)

**Purpose:** Define what each metric means and how it is computed.

Per metric:
- `metric_id`
- `domain`
- `source_of_truth` (Firestore collection / finance file family / SAP endpoint)
- `slot_schema` (period, entity, category, department)
- `normalization_rules` (typos, aliases, canonical labels)
- `calculation_rule` (row/column matching + aggregation)
- `output_contract` (value, unit, sign convention, confidence policy)

**Rule:** If a metric is not in catalog, answer can only be exploratory with explicit low confidence.

---

## 2) Query Planner

**Purpose:** Compile natural language into an executable plan.

Plan structure:
- `query_id`
- `intent`
- `metric_id`
- `resolved_slots`
- `execution_steps[]`
- `fallback_policy`

**Planner constraints:**
- no direct free-form value generation,
- must choose a known `metric_id` or explicit fallback path,
- must emit why this source was chosen.

---

## 3) Deterministic Executors

**Purpose:** Execute plans against data sources with predictable output.

Executor types:
- Firestore executor
- Finance CSV executor
- SAP executor (future-ready)

Each executor returns:
- `value`
- `source_ref` (collection/file/query segment)
- `calc_details` (matched rows/columns, aggregation mode)
- `confidence`
- `data_quality_flags`

---

## 4) Answer Renderer

**Purpose:** Convert deterministic result into user-facing response.

Must always include:
- result value,
- period/entity context,
- source summary,
- confidence or warning.

Renderer never invents numbers.

---

## 5) Observability & Replay

For each query:
- raw question,
- planner output,
- executor inputs/outputs,
- rendered answer,
- trace id.

Replay endpoint must reconstruct answer from stored trace.

---

## 6) Quality Gates

Release gates:
- DoD checks pass,
- golden business tests pass,
- no critical drift in recent traces,
- dictionary coverage threshold maintained.

---

## 7) Learning Loop (Controlled)

Data:
- `chat-traces.jsonl`
- `chat-feedback.jsonl`

Usage:
- improve intent/slot extraction,
- suggest catalog extensions,
- never auto-publish calculation changes without tests.

---

## Migration Strategy

1. Build Semantic Layer for top 20 recurring business metrics.
2. Route those metrics through planner + deterministic executors only.
3. Keep legacy tool flow as fallback with explicit warning.
4. Increase catalog coverage domain by domain.
5. Decommission legacy ambiguous paths after parity.
